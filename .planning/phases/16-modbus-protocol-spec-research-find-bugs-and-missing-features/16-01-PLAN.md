  @override

    if (functionCode.type == FunctionType.read) {

  <name>Task 1: Add response byte count and unit ID validation (BUG-02 + BUG-03)</name>
  <files>packages/modbus_client/lib/src/modbus_request.dart, packages/modbus_client_tcp/lib/src/modbus_client_tcp.dart, packages/modbus_client_tcp/test/modbus_client_tcp_test.dart</files>
  <behavior>
    - BUG-02 Test 1: FC03 response with correct byte count (N*2 for N registers) is accepted
    - BUG-02 Test 2: FC03 response with byte count too small (e.g., 2 when reading 2 registers = expected 4) returns requestRxFailed
    - BUG-02 Test 3: FC01 response with correct byte count (ceil(N/8) for N coils) is accepted
    - BUG-02 Test 4: FC01 response with wrong byte count returns requestRxFailed
    - BUG-03 Test 1: Response with matching unit ID is accepted normally
    - BUG-03 Test 2: Response with mismatched unit ID returns requestRxFailed
  </behavior>
  <action>
    **BUG-02 (modbus_request.dart):** In `ModbusElementRequest.internalSetFromPduResponse`, for read responses (`functionCode.type == FunctionType.read`), validate that `pdu[1]` (the byte count field) matches the expected size before calling `internalSetElementData(pdu.sublist(2))`. The expected byte count depends on the element's `byteCount` property. For register reads (FC03/FC04), expected = element.byteCount. For coil/discrete reads (FC01/FC02), expected = ceil(quantity/8) but since we read 1 element, expected = 1. If the byte count does not match, return `ModbusResponseCode.requestRxFailed` instead of `requestSucceed`. Log a warning via `ModbusAppLogger.warning`.

    Note: The element's `byteCount` field already holds the expected response byte count (e.g., 2 for uint16, 4 for int32, 1 for bit). Use this directly for validation. For `ModbusGroupRequest` (batch reads), byte count validation is more complex -- skip validation for group requests (their `internalSetFromPduResponse` override handles data differently).

    **BUG-03 (modbus_client_tcp.dart):** In `_TcpResponse.addResponseData`, after validating transaction ID and protocol ID (around line 402, after `_resDataLen` is set), add a unit ID check. The request's unit ID is available via `request.unitId`. Compare `_data[6]` (unit ID byte in MBAP header) against `request.unitId`. If they don't match, log a warning and set `ModbusResponseCode.requestRxFailed`. Note: `request.unitId` defaults to 1 but is set per-request. The _TcpResponse already has access to `request`. Add the check right after the `_resDataLen` range validation (line ~413) but before the "got all data" check (line ~418).
  </action>
