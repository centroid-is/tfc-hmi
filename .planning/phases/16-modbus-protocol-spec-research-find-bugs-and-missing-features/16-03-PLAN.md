<objective>
Add byte/word order configuration per Modbus server (FEAT-01), wiring the existing `ModbusEndianness` enum from the modbus_client library through the config, wrapper, and UI layers. This is the single most common Modbus interoperability issue in the field -- different device manufacturers use different byte orderings for multi-register types.

Purpose: Enable operators to connect to devices that use non-standard byte ordering (CDAB word swap, BADC byte swap, DCBA word+byte swap) without code changes.
Output: Per-server byte order dropdown in server config, endianness passed through to all element reads and writes.
</objective>

  <files>packages/tfc_dart/lib/core/state_man.dart, packages/tfc_dart/lib/core/modbus_client_wrapper.dart, packages/tfc_dart/test/core/modbus_client_wrapper_test.dart</files>
  <behavior>
    - Test 1: ModbusRegisterSpec with endianness=CDAB creates element with CDAB endianness
    - Test 2: ModbusRegisterSpec with default endianness creates element with ABCD
    - Test 3: Endianness only affects multi-register types (int32, uint32, float32, int64, uint64, float64); single-register (int16, uint16) and bit types always use ABCD
    - Test 4: ModbusConfig.endianness round-trips through JSON serialization (toJson/fromJson)
    - Test 5: ModbusConfig without endianness field in JSON defaults to ABCD (backward compatibility)
    - Test 6: Write operations pass endianness through to element requests
  </behavior>
  <action>
    **1. Add endianness to ModbusConfig** (state_man.dart):
    Add field `ModbusEndianness endianness` with default `ModbusEndianness.ABCD`. Add JSON annotation: `@JsonKey(name: 'endianness', defaultValue: ModbusEndianness.ABCD)`. The `ModbusEndianness` enum is from the modbus_client library -- import it. Since `ModbusEndianness` uses uppercase names (ABCD, CDAB, etc.), JsonSerializable should handle enum serialization by name automatically. Run `dart run build_runner build` after adding the field to regenerate `_$ModbusConfigFromJson`/`_$ModbusConfigToJson`.

    Note: Endianness is per-device (per ModbusConfig), NOT per-register (not per ModbusNodeConfig). This is the common pattern -- all registers on a device typically use the same byte order.

    **2. Add endianness to ModbusRegisterSpec** (modbus_client_wrapper.dart):
    Add optional field: `final ModbusEndianness endianness;` with default `ModbusEndianness.ABCD` in the constructor.

    **3. Wire endianness in _createElement** (modbus_client_wrapper.dart):
    Pass `endianness: spec.endianness` to all multi-register element constructors (ModbusInt32Register, ModbusUint32Register, ModbusFloatRegister, ModbusInt64Register, ModbusUint64Register, ModbusDoubleRegister). Do NOT pass endianness to single-register types (ModbusInt16Register, ModbusUint16Register) or bit types (ModbusCoil, ModbusDiscreteInput) -- these are single-byte/single-register and endianness doesn't apply.

    **4. Wire endianness in spec construction** (state_man.dart):
    In `buildSpecsFromKeyMappings` (or wherever `ModbusRegisterSpec` is constructed from `ModbusNodeConfig` + `ModbusConfig`), pass the server's `ModbusConfig.endianness` to the `ModbusRegisterSpec` constructor. Find where specs are built -- look for `ModbusRegisterSpec(` calls and add `endianness: config.endianness` where `config` is the matching `ModbusConfig` for the server alias.

    **5. Regenerate JSON serialization:**
    ```bash
    cd packages/tfc_dart && dart run build_runner build --delete-conflicting-outputs
    ```

    Import `ModbusEndianness` from the modbus_client package in state_man.dart. It may already be available through existing imports of modbus_client types.
  </action>
