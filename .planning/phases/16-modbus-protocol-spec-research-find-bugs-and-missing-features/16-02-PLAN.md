    - FEAT-03 Test 1: write() failure includes exception code name AND numeric code in error message
    - FEAT-03 Test 2: writeMultiple() failure includes exception code name AND numeric code in error message

  <action>
    **BUG-01 -- Address validation at three layers:**

    1. **ModbusRegisterSpec constructor** (modbus_client_wrapper.dart line ~39): Add assertion `assert(address >= 0 && address <= 65535, 'Modbus address must be 0-65535, got: $address')`.

    2. **ModbusNodeConfig** (state_man.dart line ~302): The constructor doesn't currently validate. Add an assertion: `assert(address >= 0 && address <= 65535, 'Modbus address must be 0-65535')` or clamp in the constructor body. Since this is a JSON-deserialized config class, prefer clamping over assertion to avoid crashes on bad stored data: add `address = address.clamp(0, 65535)` in the constructor body (after the initializer).

    3. **Key repository UI** (key_repository.dart line ~1482): Change `address: int.tryParse(_addressController.text) ?? 0` to `address: (int.tryParse(_addressController.text) ?? 0).clamp(0, 65535)`. This prevents the UI from ever creating an out-of-range address.

    **FEAT-03 -- Rich write error messages:**

    In modbus_client_wrapper.dart, update the two error throws:

    Line 379 (write): Change `throw StateError('Write failed: ${result.name}')` to include the numeric code and a human-readable description:
    ```dart
    throw StateError('Write failed: ${result.name} (code: ${result.code}) -- ${_describeException(result)}');
    ```

    Line 405 (writeMultiple): Same pattern:
    ```dart
    throw StateError('Write multiple failed: ${result.name} (code: ${result.code}) -- ${_describeException(result)}');
    ```

    Add a private helper `_describeException(ModbusResponseCode code)` that returns human-readable strings for the standard Modbus exception codes:
    - 0x01 illegalFunction: "Function code not supported by device"
    - 0x02 illegalDataAddress: "Register address does not exist on device"
    - 0x03 illegalDataValue: "Value out of range for this register"
    - 0x04 deviceFailure: "Device internal error"
    - 0x05 acknowledge: "Request accepted but processing not complete"
    - 0x06 deviceBusy: "Device busy, retry later"
    - 0x0A gatewayPathUnavailable: "Gateway path unavailable"
    - 0x0B gatewayTargetNoResponse: "Gateway target device not responding"
    - default: "Unexpected error"

    Check `ModbusResponseCode` enum for the `.code` getter -- if it doesn't exist, use the `.index` or match by name. Look at the enum definition to determine how to get the numeric code.
  </action>

    - Test 3: Unit ID 247 is accepted (was max before, still valid)
    - Test 4: Unit ID 256 is clamped to 255
    - Test 5: Unit ID -1 is clamped to 0
  </behavior>
  <action>
    **Server config UI** (server_config.dart line 1618): Change `.clamp(1, 247)` to `.clamp(0, 255)`.

    Per Modbus TCP Implementation Guide:
    - Unit ID 0 = broadcast (no response expected). Valid but rarely used in TCP.
    - Unit ID 1-247 = standard device addressing.
    - Unit ID 248-254 = reserved in serial spec but valid in TCP.
    - Unit ID 255 = commonly used for direct IP-addressed TCP devices.

    No warning for unit ID 0 or 248-254 is needed -- these are valid TCP values and a warning would be confusing for users who intentionally use them.

    Also update any tooltip or label text near the unit ID field if it mentions "1-247" to say "0-255".

    **ModbusConfig** (state_man.dart): The constructor defaults to `unitId = 1` which is correct. No assertion needed since the UI enforces the range. But for safety, add a clamp in the constructor body: `unitId = unitId.clamp(0, 255)` (same pattern as address validation above, defensive against bad stored JSON).

    Write tests in the wrapper test file to verify ModbusConfig accepts unit IDs 0, 255, and clamps out-of-range values.
  </action>
