<objective>
Add concurrent request support to ModbusClientTcp using a transaction ID map, enabling multiple in-flight requests to the same device. This is the most complex fix in Phase 1, changing the data flow from single-response to multi-response routing.

Purpose: Without this fix, all Modbus requests are fully serialized -- request B cannot start until request A's response is received. This prevents pipelining and limits throughput to one round-trip per request. With the fix, multiple requests can be in-flight simultaneously, each resolving independently via their MBAP transaction ID.

Output: Updated ModbusClientTcp with `_pendingResponses` map, narrowed lock scope, and response routing by transaction ID. Full test coverage for concurrent scenarios.
</objective>

    1. Replace `_TcpResponse? _currentResponse` with `final Map<int, _TcpResponse> _pendingResponses = {}`.

    2. Restructure `send()` to narrow the lock scope:
    ```dart
    Future<ModbusResponseCode> send(ModbusRequest request) async {
      int transactionId;
      transactionId = await _lock.synchronized(() async {
        // Connect if needed
        try {
          if (connectionMode != ModbusConnectionMode.doNotConnect) {
            await connect();
          }
          if (!isConnected) {
            return -1;  // Signal connection failure
          }
        } catch (ex) {
          ModbusAppLogger.severe("Unexpected exception in sending TCP message", ex);
          return -1;
        }

        // Flush old pending data
        await _socket!.flush();

        // Create the new response handler
        var tid = _getNextTransactionId();
        var response = _TcpResponse(request,
            transactionId: tid, timeout: getResponseTimeout(request));
        _pendingResponses[tid] = response;

        // Reset request
        request.reset();

        // Build and send MBAP frame
        int pduLen = request.protocolDataUnit.length;
        var header = Uint8List(pduLen + 7);
        ByteData.view(header.buffer)
          ..setUint16(0, tid)
          ..setUint16(2, 0)
          ..setUint16(4, pduLen + 1)
          ..setUint8(6, getUnitId(request));
        header.setAll(7, request.protocolDataUnit);
        _socket!.add(header);
        ModbusAppLogger.finest("Sent data: ${ModbusAppLogger.toHex(header)}");

        return tid;
      });

      // Connection failed?
      if (transactionId == -1) {
        return ModbusResponseCode.connectionFailed;
      }

      // Wait for response OUTSIDE the lock
      var res = await request.responseCode;
      _pendingResponses.remove(transactionId);

      // Need to disconnect?
      if (connectionMode == ModbusConnectionMode.autoConnectAndDisconnect) {
        await disconnect();
      }
      return res;
    }
    ```

    3. Rewrite `_onSocketData()` to route by transaction ID and handle concatenated responses:
    ```dart
    void _onSocketData(Uint8List data) {
      // We need to handle:
      // a) Routing to correct _TcpResponse by transaction ID
      // b) Concatenated responses (remaining bytes after one completes)
      // c) Partial responses (buffer until complete)

      // Use a buffer for unrouted data
      _incomingBuffer += data;
      _processIncomingBuffer();
    }
    ```

    Add a class-level `List<int> _incomingBuffer = []` field and a `_processIncomingBuffer()` method that:
    - Loops while buffer has >= 6 bytes (minimum to read MBAP header)
    - Reads transaction ID from bytes 0-1
    - Reads length field from bytes 4-5
    - Validates length field (1-254 per TCPFIX-03)
    - Computes total frame size = length + 6
    - If buffer has enough bytes for full frame:
      - Look up `_pendingResponses[transactionId]`
      - If found: call `addResponseData` with the full frame bytes
      - If not found: log warning "Response for unknown transaction ID {tid}, discarding"
      - Remove consumed bytes from buffer, continue loop
    - If buffer doesn't have enough bytes: break (wait for more data)

    This approach moves the MBAP parsing out of `_TcpResponse.addResponseData` and into `_onSocketData` (now the router), which is cleaner for multi-response handling. `_TcpResponse.addResponseData` can be simplified to just receive a complete MBAP frame.

    **IMPORTANT:** Keep `_TcpResponse.addResponseData` working for the case where it receives a complete frame. The TCPFIX-01 fix (`_data.length >= _resDataLen! + 6`) and TCPFIX-03 validation should remain in `_TcpResponse` as a defense-in-depth check, even though the router also validates.

    4. Update `disconnect()` to clear `_pendingResponses` and `_incomingBuffer`:

    ```dart
        _socket!.destroy();
        _socket = null;
      }
      _pendingResponses.clear();
      _incomingBuffer = [];
    }
    ```
