# Phase 8: Config Serialization - Research

**Researched:** 2026-03-06
**Domain:** JSON serialization of Modbus config models integrated into existing StateManConfig/KeyMappingEntry structures
**Confidence:** HIGH

## Summary

Phase 8 adds two new JSON-serializable config classes (ModbusConfig and ModbusNodeConfig) and integrates them into the existing StateManConfig and KeyMappingEntry structures. The project already uses `json_annotation` + `json_serializable` + `build_runner` for all config serialization in `state_man.dart` with generated code in `state_man.g.dart`. The M2400 protocol integration (Phase 4-era) established the exact pattern to follow: M2400Config was added to StateManConfig with `@JsonKey(defaultValue: [])` for backward compatibility, and M2400NodeConfig was added as a nullable field on KeyMappingEntry. The Modbus config must follow this identical pattern.

A key serialization challenge is that `ModbusElementType` from the `modbus_client` package is a class (not a Dart enum), so it cannot be directly serialized by `json_serializable`. The solution is to create a parallel `ModbusRegisterType` enum in our own code that maps to `ModbusElementType` values, similar to how `ModbusDataType` already exists as a Dart enum in `modbus_client_wrapper.dart`. This enum can then be serialized natively by json_serializable.

The `createModbusDeviceClients` factory in `modbus_device_client.dart` currently takes an anonymous record type. Phase 8 will replace this with proper `ModbusConfig` + `ModbusNodeConfig` classes that serialize cleanly and feed into the factory.

**Primary recommendation:** Follow the established M2400Config/M2400NodeConfig pattern exactly -- add ModbusConfig to StateManConfig with `@JsonKey(defaultValue: [])` and ModbusNodeConfig as a nullable field on KeyMappingEntry, using json_serializable code generation.

### Pattern 1: Server Config with Backward-Compatible Default
**What:** Add a new protocol server list to StateManConfig with `@JsonKey(defaultValue: [])` so existing JSON files without the new field parse successfully.
**When to use:** Adding a new protocol to the multi-protocol config.
**Example:**
```dart
// Source: packages/tfc_dart/lib/core/state_man.dart (existing M2400 pattern)
@JsonSerializable(explicitToJson: true)
class StateManConfig {
  List<OpcUAConfig> opcua;
  @JsonKey(defaultValue: [])
  List<M2400Config> jbtm;
  @JsonKey(defaultValue: [])       // NEW -- backward compatible
  List<ModbusConfig> modbus;       // NEW

  StateManConfig({
    required this.opcua,
    this.jbtm = const [],
    this.modbus = const [],        // NEW
  });
  // ... existing methods unchanged
}
```

### Pattern 2: Node Config as Nullable Field on KeyMappingEntry
**What:** Add a new nullable node config field to KeyMappingEntry so existing entries (which lack the field) parse with null.
**When to use:** Adding a new protocol's per-key config alongside existing protocols.
**Example:**
```dart
// Source: packages/tfc_dart/lib/core/state_man.dart (existing M2400 pattern)
@JsonSerializable(explicitToJson: true)
class KeyMappingEntry {
  @JsonKey(name: 'opcua_node')
  OpcUANodeConfig? opcuaNode;
  @JsonKey(name: 'm2400_node')
  M2400NodeConfig? m2400Node;
  @JsonKey(name: 'modbus_node')    // NEW
  ModbusNodeConfig? modbusNode;    // NEW
  bool? io;
  CollectEntry? collect;

  String? get server =>
      opcuaNode?.serverAlias ??
      m2400Node?.serverAlias ??
      modbusNode?.serverAlias;     // NEW fallback
  // ...
}
```

### Pattern 3: Enum-Based Register Type for Serialization
**What:** Create a `ModbusRegisterType` Dart enum that maps to `ModbusElementType` class instances, enabling json_serializable to handle serialization natively.
**When to use:** When the upstream library uses a class instead of a Dart enum for a type that needs JSON serialization.
**Example:**
```dart
/// Dart enum for JSON serialization -- maps to modbus_client's ModbusElementType class.
enum ModbusRegisterType {
  coil,
  discreteInput,
  holdingRegister,
  inputRegister;

  /// Convert to modbus_client's ModbusElementType for runtime use.
  ModbusElementType toModbusElementType() {
    switch (this) {
      case ModbusRegisterType.coil:
        return ModbusElementType.coil;
      case ModbusRegisterType.discreteInput:
        return ModbusElementType.discreteInput;
      case ModbusRegisterType.holdingRegister:
        return ModbusElementType.holdingRegister;
      case ModbusRegisterType.inputRegister:
        return ModbusElementType.inputRegister;
    }
  }

  /// Convert from modbus_client's ModbusElementType.
  static ModbusRegisterType fromModbusElementType(ModbusElementType type) {
    if (type == ModbusElementType.coil) return ModbusRegisterType.coil;
    if (type == ModbusElementType.discreteInput) return ModbusRegisterType.discreteInput;
    if (type == ModbusElementType.holdingRegister) return ModbusRegisterType.holdingRegister;
    if (type == ModbusElementType.inputRegister) return ModbusRegisterType.inputRegister;
    throw ArgumentError('Unknown ModbusElementType: $type');
  }
}
```

### Pattern 4: Poll Group Config Serialization
**What:** Serialize poll group name + interval as part of ModbusConfig so the server's poll groups are persisted.
**When to use:** When runtime poll configuration needs to survive app restarts.
**Example:**
```dart
@JsonSerializable(explicitToJson: true)
class ModbusPollGroupConfig {
  final String name;
  @JsonKey(name: 'interval_ms')
  final int intervalMs;

  const ModbusPollGroupConfig({
    required this.name,
    this.intervalMs = 1000,
  });

  Duration get interval => Duration(milliseconds: intervalMs);

  factory ModbusPollGroupConfig.fromJson(Map<String, dynamic> json) =>
      _$ModbusPollGroupConfigFromJson(json);
  Map<String, dynamic> toJson() => _$ModbusPollGroupConfigToJson(this);
}
```

### Anti-Patterns to Avoid
- **Using ModbusElementType directly in JSON:** It is a class, not a Dart enum. json_serializable cannot auto-serialize it. Use ModbusRegisterType enum instead.
- **Storing Duration directly:** json_serializable does not handle Duration natively. Store as `int intervalMs` and provide a `Duration get interval` getter (or use a custom converter like existing DurationMicrosecondsConverter).
- **Breaking existing JSON:** Never make previously optional fields required. Always use `defaultValue` or nullable types for new fields.
- **Modifying state_man.g.dart by hand:** Always regenerate with `dart run build_runner build`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON serialization | Manual fromJson/toJson maps | json_serializable code gen | Consistency with codebase, less error-prone, handles nullability |
| Enum string serialization | Custom string mapping | json_serializable's `$enumDecode` | Already proven in M2400RecordType/M2400Field patterns |
| Backward compatibility | Version checking logic | `@JsonKey(defaultValue: [])` | json_serializable handles missing keys automatically |
| Config file loading | Custom parsers | Existing `StateManConfig.fromFile()` / `fromPrefs()` | Already handles the full lifecycle |

**Key insight:** The codebase already solved every serialization pattern needed for this phase when M2400 was integrated. The only new thing is the ModbusElementType-to-enum mapping.

## Common Pitfalls

### Pitfall 1: Forgetting to Run build_runner After Modifying Annotated Classes
**What goes wrong:** Tests fail with compilation errors because `_$ModbusConfigFromJson` does not exist in `state_man.g.dart`.
**Why it happens:** Adding new `@JsonSerializable` classes requires regenerating the `.g.dart` file.
**How to avoid:** Run `dart run build_runner build --delete-conflicting-outputs` in `packages/tfc_dart/` after every change to annotated classes.
**Warning signs:** Compile errors referencing `_$` prefixed functions.

### Pitfall 2: Breaking the `server` Getter Precedence
**What goes wrong:** KeyMappingEntry.server returns wrong alias when multiple protocol nodes are set.
**Why it happens:** The `server` getter uses `??` fallback chain. Adding modbusNode must go at the end.
**How to avoid:** Keep OPC UA first (current behavior), then M2400, then Modbus: `opcuaNode?.serverAlias ?? m2400Node?.serverAlias ?? modbusNode?.serverAlias`.
**Warning signs:** Existing tests for `server` getter fail.

### Pitfall 3: KeyMappings.lookupServerAlias Not Updated
**What goes wrong:** Modbus keys return null server alias from `lookupServerAlias()`.
**Why it happens:** `lookupServerAlias` currently only checks `opcuaNode` and `m2400Node`.
**How to avoid:** Update `lookupServerAlias` to also check `modbusNode?.serverAlias`.
**Warning signs:** Phase 9 integration fails to route Modbus keys to correct device client.

### Pitfall 4: ModbusDataType vs ModbusRegisterType Confusion
**What goes wrong:** Using the wrong enum in the config class.
**Why it happens:** Both relate to Modbus typing but serve different purposes.
**How to avoid:** ModbusDataType = value interpretation (int16, float32, etc). ModbusRegisterType = register type (coil, holding, etc). ModbusNodeConfig needs BOTH.
**Warning signs:** Config has register type but no data type, or vice versa.

### Pitfall 5: explicitToJson Missing on New Classes
**What goes wrong:** Nested objects serialize as `Instance of 'ModbusConfig'` instead of JSON maps.
**Why it happens:** Without `explicitToJson: true`, json_serializable calls `toString()` on nested objects.
**How to avoid:** Always use `@JsonSerializable(explicitToJson: true)` -- every existing class in this codebase uses it.
**Warning signs:** JSON output contains "Instance of" strings instead of nested maps.

## Open Questions

1. **Should ModbusConfig include `type` field like M2400Config?**
   - What we know: M2400Config has `@JsonKey(defaultValue: 'm2400') String type;` for identification
   - What's unclear: Whether this is needed for Modbus or if the presence of `modbus` key in StateManConfig is sufficient
   - Recommendation: Include `type` field defaulting to `'modbus_tcp'` for consistency and future RTU support (v2)

2. **Where to place ModbusRegisterType enum?**
   - What we know: ModbusDataType is in modbus_client_wrapper.dart. Config classes are in state_man.dart.
   - What's unclear: Whether to co-locate with ModbusDataType or with config classes
   - Recommendation: Place in state_man.dart alongside ModbusNodeConfig since it's a serialization concern. Import ModbusElementType for the conversion methods.
