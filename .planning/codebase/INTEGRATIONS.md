# External Integrations

**Analysis Date:** 2026-05-05

## Industrial Protocols

**OPC UA (IEC 62541):**
- Purpose: Primary real-time data acquisition from PLCs and industrial controllers
- SDK/Client: `open62541` — C library with Dart FFI bindings (git: `github.com/centroid-is/open62541_dart`, ref: `main`); compiled via Dart native build hooks at `dart pub get` / `dart build` time
- State machine: `packages/tfc_dart/lib/core/state_man.dart` — `StateManClient` manages OPC UA subscriptions with reconnect/retry, SSL cert auth, data-change callbacks into RxDart streams
- Config: `stateman.json` file (path via `CENTROID_STATEMAN_FILE_PATH` env var); JSON-serialized `StateManConfig` with `opcua[]` array containing endpoint URL, node IDs, SSL cert/key paths, username/password
- Cert generation: `packages/tfc_dart/bin/generate_certs.dart` — creates self-signed X.509 certs via `basic_utils`
- UI: `lib/widgets/browse_panel.dart`, `lib/widgets/umas_browse.dart`, `test/widgets/opcua_browse_test.dart`

**Modbus TCP:**
- Purpose: Communication with Schneider PLCs via Modbus/TCP (also used for UMAS protocol over Modbus FC90)
- SDK/Client: Local fork of `cabbi/modbus_client` + `cabbi/modbus_client_tcp` at `packages/modbus_client/` and `packages/modbus_client_tcp/` — includes frame-parsing and keepalive fixes not in upstream
- Wrapper: `packages/tfc_dart/lib/core/modbus_client_wrapper.dart` — `ModbusClientWrapper` wraps the TCP client with `ModbusRegisterSpec` poll groups and RxDart streams
- Device adapter: `packages/tfc_dart/lib/core/modbus_device_client.dart` — `ModbusDeviceClientAdapter`

**UMAS Protocol (Schneider FC90):**
- Purpose: Read variable dictionaries and live values from Schneider PLCs (e.g., M340, M580) using Modbus function code 0x5A (FC90) as transport
- Implementation: `packages/tfc_dart/lib/core/umas_client.dart` — `UmasClient` sends `UmasRequest` (subclasses `ModbusRequest`) and parses data dictionary (DD02 records)
- Types: `packages/tfc_dart/lib/core/umas_types.dart` — `UmasVariable`, `UmasSubFunction` enum, `mapUmasDataTypeToModbus()`
- UMAS sub-functions: init (0x01), readId (0x02), readProjectInfo (0x03), readDataDictionary (0x26)
- UI: `lib/widgets/umas_browse.dart`
- Tests: `packages/tfc_dart/test/core/umas_client_test.dart`, `packages/tfc_dart/test/umas_live_test.dart`, `packages/tfc_dart/test/umas_e2e_test.dart`

**M2400 / JBTM Protocol:**
- Purpose: Data acquisition from Marel M2400 weighing devices (proprietary TCP binary protocol)
- Package: `packages/jbtm/` — `M2400ClientWrapper` (`lib/src/m2400_client_wrapper.dart`), `M2400Proxy` binary (`bin/m2400_proxy.dart`), record types: `recWgt`, `recIntro`, `recStat`, `recLua`, `recBatch`
- Transport: Custom TCP socket (`lib/src/msocket.dart`) with connection health monitoring
- Backend deployment: `m2400_proxy` binary compiled and deployed inside `ghcr.io/centroid-is/centroid-backend` Docker image

## Data Storage

**Databases:**
- **TimescaleDB** (PostgreSQL 17 + timescaledb extension) — Production time-series database for telemetry, alarms, and HMI configuration
  - Connection: env vars `CENTROID_PGHOST`, `CENTROID_PGPORT`, `CENTROID_PGDATABASE`, `CENTROID_PGUSER`, `CENTROID_PGPASSWORD`, `CENTROID_PGSSLMODE`
  - TLS: SSL required in production (self-signed certs generated by `pg-certs` Docker init container)
  - Hypertables: `database_drift.dart` line 857: `SELECT create_hypertable(...)` for time-series telemetry tables; retention policies via `timescaledb_information.jobs`
  - Docker image: `timescale/timescaledb:latest-pg17`
  - ORM client: Drift ^2.28.0 + drift_postgres ^1.3.1; custom PostgreSQL driver fork at `github.com/centroid-is/postgresql-dart` (branch `add-keepalive-test`) via `dependency_overrides` in all packages

- **SQLite** (local, via Drift `drift_flutter ^0.2.4`) — Offline/local storage; same Drift schema compiles to both SQLite and PostgreSQL dialects (controlled by `build.yaml`)

- **SharedPreferences** (`shared_preferences ^2.3.5`) — Lightweight key-value store for page layout, menu config, and non-sensitive preferences

**File Storage:**
- Local filesystem: OPC UA certificates (`/certs/` in Docker), state manager config JSON, log files, PDF tech docs
- No cloud file storage

**Caching:**
- None (no Redis or in-memory cache layer)

## LLM / AI Providers

**Claude (Anthropic):**
- SDK: `anthropic_sdk_dart ^1.2.0`
- Provider impl: `lib/llm/claude_provider.dart`
- Auth: API key stored in secure keyring under preference key `llm.claude.api_key`; optional custom base URL at `llm.claude.base_url`
- Used by: `lib/providers/llm.dart`, `lib/providers/chat.dart`

**OpenAI:**
- SDK: `openai_dart ^1.1.0`
- Provider impl: `lib/llm/openai_provider.dart`
- Auth: API key at `llm.openai.api_key`; optional custom base URL at `llm.openai.base_url`

**Gemini (Google):**
- No dedicated SDK; HTTP-based via `http ^1.6.0`
- Provider impl: `lib/llm/gemini_provider.dart`
- Auth: API key at `llm.gemini.api_key`; no custom base URL support

**MCP (Model Context Protocol):**
- Package: `mcp_dart ^2.0.0`
- Server: `packages/tfc_mcp_server/` — exposes HMI state (alarm data, trend data, config) as MCP tools for AI assistants
- Bridge: `lib/mcp/mcp_bridge_notifier.dart` — manages MCP server lifecycle; SSE transport via `lib/mcp/mcp_sse_server.dart`
- State reader: `lib/mcp/state_man_state_reader.dart`, `lib/mcp/alarm_man_alarm_reader.dart`
- Elicitation: `lib/mcp/mcp_bridge_notifier.dart` — write-tool proposals surface confirmation dialogs via `ElicitationDialog`

## Authentication & Identity

**OPC UA Security:**
- Certificate-based mutual TLS authentication between HMI and OPC UA servers
- X.509 certs generated by `packages/tfc_dart/bin/generate_certs.dart`
- Username/password auth also supported (configurable per server in `stateman.json`)

**Secure Storage:**
- Linux: `libsecret` (GNOME Keyring / KWallet) via `amplify_secure_storage_dart ^0.5.6`; fallback in Docker: file-based keyring at path from `SECRET_FILE_TEST_PATH`
- Windows: Custom `OtherSecureStorage` (set in `centroid-hmi/lib/main.dart` at startup for Windows platform)
- macOS: System Keychain via `amplify_secure_storage_dart`
- Stores: LLM API keys, PostgreSQL credentials, OPC UA passwords

**LLM API Keys:**
- Stored in secure keyring, read via `packages/tfc_dart/lib/core/preferences.dart` `getString(key, secret: true)`
- Selected provider persisted at `llm.selected_provider` in regular preferences

## System Integration (Linux)

**DBus:**
- Package: `dbus ^0.7.11`
- Usage: `lib/dbus/ipc-ruler.dart`, `lib/dbus/config.dart`, `lib/dbus/remote.dart`, `lib/providers/dbus.dart`
- Providers: `lib/dbus/generated/` — generated DBus interface stubs
- Auth: Login form at `lib/pages/dbus_login.dart` to authenticate DBus session

**NetworkManager:**
- Package: `nm ^0.5.0`
- Usage: `lib/pages/ip_settings.dart` — IP settings page (Linux only)
- Requires authenticated DBus session

**SSH:**
- Package: `dartssh2 ^2.11.0`
- Usage: `lib/dbus/remote.dart` — SSH tunneling for remote DBus access

## Monitoring & Observability

**Error Tracking:**
- No external service (Sentry, Datadog, etc.)

**Logs:**
- Structured logging via `logger ^2.4.0` throughout Dart codebase
- Log level configured by `CENTROID_LOG_LEVEL` env var
- OPC UA log level by `CENTROID_OPCUA_LOG_LEVEL`
- In Docker/production: log to file via `CENTROID_LOG_FILE` or stdout via `CENTROID_STDOUT`
- MSIX (Windows): Synchronous `RandomAccessFile` log writer for MSIX sandbox environment

## CI/CD & Deployment

**Hosting:**
- **Linux (elinux)**: Docker image `ghcr.io/centroid-is/centroid-hmi:{sha|latest|stable}` on GHCR; deployed via `docker-compose.yml` with Weston/Wayland compositor
- **Backend**: Docker image `ghcr.io/centroid-is/centroid-backend:{sha|latest|stable}` on GHCR
- **macOS**: Signed + notarized `.dmg` artifact uploaded to GitHub Actions
- **Windows**: Signed `.msix` artifact uploaded to GitHub Actions; sideload distribution

**CI Pipeline (GitHub Actions):**
- `.github/workflows/centroid-hmi.yml` — elinux build → Docker push to GHCR
- `.github/workflows/macos.yml` — macOS Flutter build → Apple code signing → notarization → DMG
- `.github/workflows/windows.yml` — Go manager build → Flutter Windows build → MSIX signing
- `.github/workflows/build-manager.yml` — `centroidx-manager` Go binary (Linux/macOS/Windows) with cross-platform integration tests
- `.github/workflows/test.yml` — Flutter tests (ubuntu/macOS/Windows matrix), modbus lib tests, tfc_dart tests with TimescaleDB
- `.github/workflows/tag.yml` — Release workflow triggered by tags
- Toolchain image: `ghcr.io/centroid-is/tfc-toolchain:latest` (custom image with `flutter-elinux`)
- Live update: `containrrr/watchtower` Docker container polls for image updates every 60s

**Release / Update System:**
- `packages/centroidx_upgrader/` — checks `github.com/centroid-is/tfc-hmi` GitHub Releases API for new versions
- `tools/centroidx-manager/` — Go native binary (Gio UI) that downloads and installs new CentroidX releases; bundled into Flutter assets at `centroid-hmi/assets/manager/`
- Update flow: `upgrader` package shows dialog → `ManagerLauncher` extracts and launches `centroidx-manager` → manager downloads new release → restarts CentroidX

## Webhooks & Callbacks

**Incoming:**

**Outgoing:**
- None (no webhooks to external services)

## Environment Configuration Summary

**Required for backend (tfc_dart `main.dart`):**
- `CENTROID_PGHOST` — required (no default)
- `CENTROID_STATEMAN_FILE_PATH` — required (no default)
- `CENTROID_PGDATABASE` — optional (default: `hmi`)
- `CENTROID_PGPORT` — optional (default: `5432`)

**Required for frontend (centroid-hmi):**
- None strictly required at startup; DB credentials entered via Server Config UI page
- `TFC_GOD=true` — unlocks editor pages
- `SECRET_BACKEND=file` + `SECRET_FILE_TEST_PATH` — for headless/Docker environments without desktop keyring

**CI secrets (GitHub Actions):**
- `APPLE_CERTIFICATE_P12_BASE64`, `APPLE_CERTIFICATE_PASSWORD`, `APPLE_TEAM_ID`, `APPLE_ID`, `APPLE_ID_PASSWORD` — macOS signing/notarization
- `MSIX_CERT_PFX_BASE64`, `MSIX_CERT_PASSWORD` — Windows MSIX signing
- `GITHUB_TOKEN` — GHCR push and manager integration tests
